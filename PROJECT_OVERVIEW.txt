PROJECT SUMMARY
================================================================================
MediTrace is a full-stack healthcare platform for Pakistani clinics that combines
hospital management functionality with AI-powered document intelligence. It transforms
unstructured medical records (free-text notes, prescription PDFs, WhatsApp exports)
into structured, searchable clinical intelligence.

Core Purpose:
  - Patients: Upload medical records, ask AI health questions, check medications
  - Doctors: Query patient records semantically, get AI-assisted diagnostics
  - Admin: Manage users, monitor costs, audit system activity

Status: BSAI 4th Semester Final Project (Production-ready on Railway & Vercel)
Team: Shiv Kumar, Usama Shehzad, Mariyam, Bhumika, Husna

WHAT IS AGENTIC AI HERE?
─────────────────────────────────────────────────────────────────────────────
Agentic AI means the AI models call external tools/functions to retrieve data,
process it, and make decisions. MediTrace implements two separate agent systems:

1. PATIENT AGENT (Gemini 1.5 Flash) — Manual Tool Loop
2. DOCTOR AGENT (GPT-4o-mini) — Function Calling

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

AGENT #1: PATIENT AI (Gemini 1.5 Flash) — Manual Tool Loop
─────────────────────────────────────────────────────────────────────────────

Location: /backend/patient_ai.py
Endpoint:  POST /ai/patient/agent

How It Works:
  1. Patient submits query: "What medications am I on?"
  2. Frontend makes HTTP request to /ai/patient/agent with patient_id
  3. Backend receives query and determines needed tool(s)
  4. Manual tool loop:
     a) Call Gemini with available tools in system prompt
     b) Parse response to detect tool calls
     c) Execute tools (search_my_documents, get_my_medications, etc.)
     d) Feed results back to Gemini
     e) Repeat until Gemini returns final answer (no tool call)
  5. Frontend streams responses via SSE, showing step-by-step indicators

Available Tools (Patient Agent):
  ┌─────────────────────────────────────────────────────────────────────┐
  │ Tool Name                  │ Purpose                                 │
  ├─────────────────────────────────────────────────────────────────────┤
  │ search_my_documents        │ Semantic search on patient's own docs   │
  │ get_my_medications         │ Get active prescriptions from DB        │
  │ triage_symptoms            │ Analyze symptoms, return urgency badge  │
  │ get_upcoming_reminders     │ Check medications due for refill        │
  └─────────────────────────────────────────────────────────────────────┘

Tool Implementation Details:

  search_my_documents(query: str, patient_id: str) → str
    ├─ Loads all embeddings for this patient's documents
    ├─ Embeds query using sentence-transformers (384-dim)
    ├─ Computes cosine similarity to all document chunks
    ├─ Returns top 3 chunks with score > 0.1 (30% threshold)
    ├─ Formats as: "From: [filename] (uploaded [date]), Chunk [N]: [text]..."
    └─ Used for: Questions like "What does my blood test say?"

  get_my_medications(patient_id: str) → str
    ├─ Query database for active prescriptions
    ├─ Format: "- [medicine] [dosage], [frequency] (#[rx_number])"
    ├─ Calculate refill date: start_date + duration_days
    ├─ Return formatted list
    └─ Used for: "What am I currently taking?"

  triage_symptoms(symptom_text: str, model: Gemini) → dict
    ├─ Send symptoms to Gemini with structured JSON prompt
    ├─ Parse JSON response:
    │   ├─ urgency: low | moderate | high | emergency
    │   ├─ suggested_action: rest at home | see doctor | go to ER
    │   ├─ possible_concerns: ["concern1", "concern2"]
    │   └─ disclaimer: Medical advice warning
    ├─ Frontend displays urgency as colored badge:
    │   ├─ LOW (green):      "Rest at home, fluids, pain relief"
    │   ├─ MODERATE (yellow):"See doctor in 2–3 days"
    │   ├─ HIGH (orange):    "Seek medical attention today"
    │   └─ EMERGENCY (red):  "Go to ER immediately or call 1122"
    └─ Used for: "I have chest pain and can't breathe"

  get_upcoming_reminders(patient_id: str) → str
    ├─ Query prescriptions ending in next 7 days
    ├─ Return: "⚠️ [medicine] — refill needed by [date]"
    └─ Used for: Proactive medication reminders

Frontend Integration (Patient Chat):
  ✓ Real-time step indicators: "Analyzing question…" → "Searching docs…" → "Generating answer…"
  ✓ Tool badges show which tools were called: [search_my_documents] [get_my_medications]
  ✓ Model badge: "Gemini 1.5 Flash"
  ✓ Disclaimer always shown: "Not medical advice. Always consult your doctor."
  ✓ Streaming response via Server-Sent Events (SSE)

Scenario Example:
  Patient: "I have a headache and fever for 2 days"
    → Tool: triage_symptoms() called
    → Response: "GREEN badge: Rest at home, drink fluids, take Paracetamol"
    → Disclaimer displayed
  
  Patient: "What medications am I on?"
    → Tool: get_my_medications() called
    → Response: Lists all active prescriptions with refill dates
    → Citation: "According to your records…"

AGENT #2: DOCTOR AI (GPT-4o-mini) — Function Calling
─────────────────────────────────────────────────────────────────────────────

Location: /backend/doctor_ai.py
Endpoint:  POST /ai/doctor/[agent_type]

How It Works:
  1. Doctor selects tab (Pharmacy, Risk, Lab, etc.) and enters patient ID
  2. Frontend sends request to specific agent endpoint
  3. Backend calls OpenAI API with function definitions (no manual loop)
  4. GPT-4o-mini automatically calls appropriate functions
  5. OpenAI handles tool orchestration; backend just implements tools
  6. Results returned with citations and confidence scores
  7. Frontend displays with step indicators + model badge

Available Agents (Doctor AI):

  ┌──────────────────────────────────────────────────────────────────────┐
  │ Agent Tab          │ Function             │ Use Case                  │

Available Tools (Doctor Agent):

  ┌──────────────────────────────────────────────────────────────────────┐
  │ Tool Name           │ Purpose                                        │
  ├──────────────────────────────────────────────────────────────────────┤
  │ search_documents    │ Semantic search with source + chunk tracking   │
  │ get_patient_history │ Full visit + prescription history             │
  │ normalize_drug_name │ Convert Pakistani brand → generic name        │
  │ get_adherence_score │ Calculate medication compliance %             │
  │ forecast_refills    │ Predict next refill dates                     │
  │ calculate_risk      │ 30-day readmission risk probability          │
  │ get_lab_results     │ Recent lab values with reference ranges       │
  │ check_contraindicate│ Drug-drug interaction checking               │
  └──────────────────────────────────────────────────────────────────────┘

Tool Implementation Details:

  search_documents(query: str, patient_id: str) → dict[]
    ├─ Similar to patient tool, but for doctor access
    ├─ Returns: [{ chunk_text, filename, chunk_index, score, upload_date }, ...]
    ├─ Citations include exact chunk numbers and upload dates
    ├─ Doctors can search across all assigned patients' documents
    └─ Example: Searching for "diabetes complications" returns relevant docs

  get_patient_history(patient_id: str) → str
    ├─ Fetch last 10 medical records (visits)
    ├─ Fetch last 10 prescriptions
    ├─ Format as markdown:
    │   ├─ ## Visits
    │   │   └─ - [date]: Symptoms: [X] | Dx: [Y]
    │   └─ ## Prescriptions
    │       └─ - [date]: [medicine] [dosage] [frequency] ([status])
    ├─ Used for: Context when analyzing patient risk/treatment
    └─ Example: Doctor queries "What's this patient's history?"

  normalize_drug_name(drug_name: str) → str
    ├─ Local dictionary lookup: "Brufen" → "Ibuprofen", "Loprin" → "Aspirin"
    ├─ ~50 common Pakistani brand names mapped to generics
    ├─ Returns generic name or None if not found
    ├─ Used for: Standardizing drug names before checking interactions
    └─ Example: Input "Augmentin", output "Amoxicillin/Clavulanate"

  get_adherence_score(patient_id: str) → dict
    ├─ Calculate medication compliance based on:
    │   ├─ Prescription fill dates vs. expected intervals
    │   ├─ Days supply vs. actual usage
    │   └─ Refill timing patterns
    ├─ Return: { score: 72%, risk_level: "medium", details: "..." }
    └─ Example: "Patient refilled 90% on schedule → adherence: 72%"

  forecast_refills(patient_id: str, days_ahead: int) → dict[]
    ├─ Calculate end dates for all active prescriptions
    ├─ Predict which will need refills in next N days
    ├─ Return: [{ medicine, last_filled, next_due, days_until }, ...]
    └─ Example: "Metformin expires in 7 days, needs refill"

  calculate_risk(patient_id: str, horizon_days: int=30) → dict
    ├─ Analyze patient history, lab values, comorbidities
    ├─ Use GPT-4 to estimate readmission probability
    ├─ Return: { risk_percentage: 15%, confidence_range: "12–18%", factors: [...] }
    ├─ Flags: Disease progression indicators
    ├─ Recommendations: Suggested investigations
    └─ Example: "35% readmission risk (high) — suggest ICU monitoring"

  check_contraindications(drug_name: str, patient_rx_list: list[str]) → dict
    ├─ Check if new drug interacts with current medications
    ├─ Query internal drug interaction database
    ├─ Return: { interactions: [...], severity: "low|moderate|high", actions: [...] }
    └─ Example: "Warfarin + Aspirin = HIGH interaction risk"

Frontend Integration (Doctor AI):

KEY AGENTIC AI FEATURES
================================================================================

1. TOOL ORCHESTRATION
   ─────────────────────────────────────────────────────────────────────────
   - Patient Agent: Manual tool loop (Gemini)
     * Backend writes prompts instructing model to call tools
     * Iterates: Gemini response → parse tools → execute → feed back
     * Continues until Gemini stops requesting tools
   
   - Doctor Agent: Native function calling (GPT-4o-mini)
     * OpenAI SDK handles orchestration automatically
     * Backend defines tools via JSON schema
     * Model calls tools; backend implements them
     * No manual loop needed

2. SEMANTIC SEARCH INTEGRATION
   ─────────────────────────────────────────────────────────────────────────
   - Both agents use semantic search to retrieve patient documents
   - Process:
     1. Query embedded using sentence-transformers (384-dim)
     2. Cosine similarity computed against all document chunks
     3. Top results (score > 0.1) returned with exact chunk numbers
     4. AI cites source with chunk number + upload date
   
   - Example: Doctor asks "What complications did this patient have?"
     → Semantic search finds relevant chunks
     → GPT-4 analyzes them
     → Returns: "Source: Discharge Summary.pdf · Chunk 3"

3. STREAMING RESPONSES
   ─────────────────────────────────────────────────────────────────────────
   - Patient chat uses Server-Sent Events (SSE) for real-time streaming
   - Frontend displays:
     * Real-time step indicators (e.g., "Analyzing question…")
     * Urgency badges (color-coded)
     * Tool calls being made
     * Streaming text as it arrives
   
   - Doctor AI: Step indicators but full result at once
     * Shows "Fetching history…" while waiting
     * Returns complete result with all citations

4. ROLE-BASED TOOL ACCESS
   ─────────────────────────────────────────────────────────────────────────
   - Patient Agent: 
     * Can only search own documents
     * Can only see own prescriptions & reminders
   
   - Doctor Agent:
     * Can search only assigned patients' documents
     * Can see full patient history (visits, labs, prescriptions)
     * Can make clinical recommendations
   
   - Admin Agent:
     * Full access to all documents and data
     * Can monitor costs, audits, system health

5. CITATIONS & EVIDENCE
   ─────────────────────────────────────────────────────────────────────────
   - Patient: "According to your [document name] uploaded on [date]…"
   - Doctor: "Source: [document.pdf] · Chunk [N] · Uploaded [date] · Confidence: [%]"
   - Every AI response includes source tracking
   - Enables auditability and verification

6. FALLBACK & ERROR HANDLING
   ─────────────────────────────────────────────────────────────────────────
   - If API key missing: Graceful degradation message
   - If tool fails: Return "Tool error - please try again"
   - If no documents found: "No relevant content found in your documents"
   - All errors logged for admin audit

================================================================================
DOCUMENT PROCESSING PIPELINE
================================================================================

How Documents Become Searchable:
─────────────────────────────────────────────────────────────────────────────

1. UPLOAD
   User uploads: discharge_summary.pdf, lab_report.csv, etc.
     ↓
   File stored in database (raw bytes in file_size field)

2. TEXT EXTRACTION
   Backend detects file type and extracts text:
     - PDF: PyMuPDF reads all pages, preserves page numbers
     - DOCX: python-docx extracts all paragraphs
     - CSV: Converts rows to "key: value" format
     - TXT: UTF-8 decode
   
   Example (PDF):
     [Page 1]
     Patient ID: 12345
     Diagnosis: Hypertension
     
     [Page 2]
     Lab Results: BP 150/90…

3. CHUNKING
   Text split into overlapping chunks:
     - Chunk size: 750 characters
     - Overlap: 100 characters (for context)
   
   Example:
     Chunk 1: [0–750 chars]
     Chunk 2: [650–1400 chars]
     Chunk 3: [1300–2050 chars]

4. EMBEDDING
   Each chunk embedded using sentence-transformers:
     - Model: all-MiniLM-L6-v2 (384 dimensions)
     - Runs locally (no API cost, no privacy concerns)
     - Vector stored in database as JSON
   
   Vector representation enables semantic similarity:
     "What complications did patient have?" 
     → Similar to chunk about diabetes complications
     → Score: 0.87 (87% similar)

5. STORAGE
   For each chunk:
     - chunk_id: UUID
     - document_id: Links to original file
     - chunk_index: Order (0, 1, 2, …)
     - chunk_text: 750-char excerpt
     - vector_json: 384-dim float array
     - user_id: Who uploaded it
   
   Enables fast semantic search:
     SELECT * FROM embeddings
     WHERE user_id = [patient_id]
     ORDER BY cosine_similarity(vector, query_vector) DESC

6. SEARCH
   When agent searches:
     - Query: "What did my blood test show?"
     - Embedded to 384-dim vector
     - Compared to all patient's chunk vectors via cosine similarity
     - Top 3 results (score > 0.1) returned
     - Agent cites: "Chunk 5 of Lab Report.pdf"

================================================================================
HYBRID SEARCH (Semantic + Keyword)
================================================================================

For general search endpoint (/search/hybrid):
─────────────────────────────────────────────────────────────────────────────

Formula: 0.4 × keyword_score + 0.6 × semantic_score

Steps:
  1. Keyword Score: Simple substring matching
     - Query: "diabetes medication"
     - Chunks containing "diabetes" OR "medication" scored higher
  
  2. Semantic Score: Cosine similarity of embeddings
     - Query embedded → compared to all chunks
     - Returns 0.0–1.0 similarity
  
  3. Combined Score: Weighted average
     - Semantic (60%) prioritized over keyword (40%)
     - Balances precision (keyword) + understanding (semantic)

Example:
  Query: "blood sugar control"
  
  Chunk A: "Patient taking Metformin 500mg for diabetes"
    Keyword: 0.5 (contains "for diabetes")
    Semantic: 0.85 (semantically similar to query)
    Final: 0.4 × 0.5 + 0.6 × 0.85 = 0.71 ✓ (High ranking)
  
  Chunk B: "Blood pressure medication for hypertension"
    Keyword: 0.3 (contains "blood")
    Semantic: 0.42 (not very similar to "blood sugar control")
    Final: 0.4 × 0.3 + 0.6 × 0.42 = 0.33 (Lower ranking)

Rate Limiting (slowapi):
─────────────────────────────────────────────────────────────────────────────
- Default: 100 requests per minute per IP
- API endpoints fully rate-limited
- Prevents abuse, protects API costs

Authentication & Authorization:
─────────────────────────────────────────────────────────────────────────────
- JWT tokens: 24-hour expiration
- Bcrypt password hashing
- Role-based access control (RBAC):
  * Patient: Only own data
  * Doctor: Only assigned patients
  * Admin: Everything
- Row-level security in queries

Input Validation:
─────────────────────────────────────────────────────────────────────────────
- Pydantic models validate all inputs
- Email regex validation
- Min/max length checks
- SQL injection protected via SQLAlchemy ORM
- HTML sanitization via bleach library

COST BREAKDOWN
================================================================================

Free / Zero Cost:
  ✓ Embeddings: sentence-transformers (local, open-source)
  ✓ Vector storage: Database (standard cost)
  ✓ Frontend: Vercel (free tier for demos)
  ✓ Database: SQLite dev / Supabase (PostgreSQL, first 5GB free)

API Costs (Scaled):
  
  Gemini 1.5 Flash (Patient Chat):
    - Free tier: 15 req/min, 1M tokens/day
    - Paid: $0.075 per 1M input tokens, $0.30 per 1M output tokens
    - Estimate: ~$0.001–0.005 per patient query
  
  GPT-4o-mini (Doctor AI):
    - Input: $0.15 per 1M tokens
    - Output: $0.60 per 1M tokens
    - Estimate: ~$0.02–0.10 per doctor query
  
  Total estimate (for 10,000 queries):
    - Patient (8,000): ~$16 (mostly free tier)
    - Doctor (2,000): ~$40 (used more tokens)
    - TOTAL: ~$56/month

Admin Cost Dashboard:
  - Tracks: API calls, tokens used, cost per model
  - Updates: Real-time (shown in /admin/costs)
  - Audit: Logged in sat_ai_interactions table

  GEMINI_API_KEY          [Google Cloud console]

  SECRET_KEY              [Random 32-char string for JWT]
  DATABASE_URL            [Supabase PostgreSQL URI]

================================================================================
UNIQUE FEATURES & HIGHLIGHTS
================================================================================

1. Pakistani-Specific Drug Normalization
   - 50+ local brand names mapped to generics
   - Example: "Brufen" → "Ibuprofen", "Loprin" → "Aspirin"

2. Dual AI Strategy
   - Patient: Free Gemini (low cost, friendly tone)
   - Doctor: GPT-4o-mini (clinical accuracy, function calling)

3. Real-Time Agent Transparency
   - Step-by-step indicators show AI reasoning
   - Tool badges show exactly what was called
   - No black box — users see what agent does

4. Zero-Trust Architecture
   - Row-level security: patients can't see others' docs
   - All queries include patient_id or doctor_id check
   - Every API call validated

5. Semantic Search + Keyword Blend
   - Hybrid formula (40/60 split) balances recall + precision
   - Enables both "find all diabetes docs" and "what this means"

6. Production-Ready Infrastructure
   - Async everywhere (no blocking)
   - PostgreSQL with asyncpg for scale
   - Vercel CDN for frontend
   - Railway for auto-scaling backend
